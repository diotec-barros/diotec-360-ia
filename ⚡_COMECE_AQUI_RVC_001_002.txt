🚨 RVC-001: FAIL-CLOSED Z3 SOLVER
═══════════════════════════════════

VULNERABILIDADE:
  Se o Z3 retornar 'unknown' ou lançar exceção, o sistema poderia aceitar
  a transação sem prova matemática.

CORREÇÃO:
  Implementado fail-closed estrito em aethel/core/judge.py:
  
  • z3.sat → PROVED ✅ (única forma de aceitar)
  • z3.unsat → FAILED ❌ (contradição detectada)
  • z3.unknown → REJECTED 🔒 (fail-closed)
  • Exception → REJECTED 🔒 (fail-closed)

PRINCÍPIO:
  "Se não podemos provar que é seguro, então é inseguro."

TESTES:
  python test_rvc_001_fail_closed_z3.py
  
  Resultado: 3 passed, 2 skipped (Z3 muito rápido para forçar 'unknown')


💰 RVC-002: DECIMAL PRECISION
═══════════════════════════════

VULNERABILIDADE:
  Uso de float em cálculos financeiros permite "Salami Attack" via erro
  de arredondamento acumulado. Atacante realiza 1M micro-transações de
  0.00000001 e rouba o "gap" criado pelo erro de arredondamento.

CORREÇÃO:
  Implementado Decimal em aethel/moe/guardian_expert.py:
  
  • Decimal com 28 dígitos de precisão (getcontext().prec = 28)
  • Igualdade EXATA (sem epsilon tolerance)
  • Validação de precisão em conversões
  • Método _parse_decimal() com validação
  • Método _validate_conservation_exact() com zero tolerância

PRINCÍPIO:
  "Zero Tolerância para Erro de Arredondamento"

TESTES:
  python test_rvc_002_decimal_precision.py
  
  Resultado: 8 passed
  
  Demonstração:
    • 1,000,000 micro-transações de 0.00000001
    • Saldo inicial: 1,000,000.00000000
    • Saldo final: 999,999.99000000 (EXATO!)
    • Diferença: 0 (zero erro de arredondamento)
